My Notes on: Paxos Made Simple

Original Paper: Paxos Made Simple by Leslie Lamport

Core Problem Addressed

The paper addresses the problem of achieving consensus among a collection of processes in a distributed system, especially in the presence of failures (non-Byzantine, such as process crashes or message loss). This is fundamental for building fault-tolerant distributed databases, replicated state machines, etc.

Key Takeaways & Concepts

  • Roles: Proposers, Acceptors, and Learners. A process can take on multiple roles.
  • Proposal Numbers: Each proposal has a unique, ordered proposal number `n`. This is crucial for distinguishing older proposals from newer ones.
  • Two Phases:
    • Phase 1 (Prepare): A proposer chooses a new proposal number `n` and sends a `prepare(n)` request to a majority of acceptors. Acceptors respond with a promise not to accept any proposal numbered less than `n`, and if they've already accepted a proposal, they include the highest-numbered proposal they've accepted.
    • Phase 2 (Accept): If the proposer receives responses from a majority, it chooses a value `v`. If any acceptors reported previously accepted values, the proposer MUST choose the value from the highest-numbered proposal among those responses. Otherwise, it can choose its own value. It then sends an `accept(n, v)` request to the acceptors. Acceptors accept the proposal `(n,v)` if they haven't promised to consider only higher-numbered proposals.
  • Learning a Chosen Value: Once a proposal is accepted by a majority of acceptors, the value is chosen. Learners find out about the chosen value.
  • Safety Properties: Only a single value can be chosen, and if a value is chosen, processes can eventually learn it.

My Summary & Thoughts

Lamport's "Paxos Made Simple" indeed makes a complex topic more digestible, though it still requires careful reading. The core idea of using proposal numbers to establish an ordering and the two-phase commit-like structure (prepare and accept) are elegant ways to ensure safety.

The distinction between choosing a value and learning a chosen value is important. The paper focuses primarily on the former. The requirement for a proposer to adopt a previously accepted value if one exists is key to preventing conflicting values from being chosen.

"The Paxos algorithm works by electing a 'leader' (the proposer whose proposal is chosen) for each round of decision-making, but it does so without a dedicated leader election phase visible to the outside." (My interpretation/summary point)

Challenges / Points of Confusion (Initial Read)

  • The concept of "liveness" (ensuring a value is eventually chosen) is more complex and often requires a distinguished proposer or leader election mechanism built on top of or integrated with Paxos to avoid dueling proposers.
  • Handling acceptor failures and re-proposals.

Connections to Other Works

This paper is foundational. Raft, for example, was designed to be more understandable than Paxos while providing similar safety guarantees, by explicitly including leader election and log replication as distinct parts of the algorithm.
